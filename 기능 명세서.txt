기능 명세서
이 페이지는 functionalSpecification 인라인 데이터베이스로 구성되어 있으며, 각 기능이 하위페이지로 등록되어 있습니다.
FNC-AUTH-01 · 인증 토큰 발급 및 검증
도메인: 공통 | 우선순위: 상 | 연관화면: 로그인 화면 | 시작 상태: 비로그인 상태 | 종료 상태: 토큰 발급 완료
기능 설명: 카카오 로그인을 통해 사용자를 식별하고 서비스 접근에 필요한 인증 토큰(Access/Refresh)을 발급 및 검증한다.
선행조건
카카오 디벨로퍼스(Kakao Developers)에 애플리케이션 등록 및 REST API 키가 백엔드 환경변수에 설정되어 있어야 한다.
오렌지파이 서버 내 사용자 정보 관리를 위한 DB(PostgreSQL)가 정상 가동 중이어야 한다.
트리거
사용자가 로그인 화면에서 '카카오 로그인' 버튼을 클릭하여 인가 코드를 서버로 전달할 때
동작
시스템은 전달받은 인가 코드로 카카오 서버에서 사용자 식별 정보를 획득한다.
DB를 조회하여 신규 사용자는 자동 가입 처리하고, 기존 사용자는 로그인 처리를 진행한다.
로그인 성공 시 서비스 자체 인증 토큰(Access Token 및 Refresh Token)을 발급한다.
Refresh Token은 서버에서 검증 가능한 형태로 안전하게 저장하고 클라이언트에게 토큰을 반환한다.
예외 / 제한사항
유효하지 않은 로그인 정보이거나 카카오 API 통신 실패 시, 토큰은 발급되지 않으며 401 Unauthorized 에러와 함께 실패 사유를 노출한다.
보안 및 서버 메모리 관리를 위해 Access Token의 만료 시간을 짧게 설정하며, 만료 시 Refresh Token을 통해 재발급하는 로직을 필수 적용한다.
비고
오렌지파이 서버 리소스를 절약하기 위해 무거운 세션(Session) 유지 방식 대신 무상태(Stateless) JWT 기반 인증 아키텍처를 채택한다.
FNC-SRC-01 · 멀티모달 질의 분석 및 통합 쿼리 생성
도메인: 공통 | 우선순위: 상 | 연관화면: 통합 검색 화면 | 시작 상태: 사용자 입력 대기 | 종료 상태: 융합 쿼리 생성 완료
기능 설명: 텍스트 질의 및 첨부된 이미지를 결합 분석하여 하이브리드 통합 검색을 위한 최종 쿼리를 생성한다.
선행조건
사용자가 정상적인 인증 토큰(Access Token)을 보유하고 로그인된 상태여야 한다.
외부 비전언어모델(VLM) API 호출을 위한 연동 세팅이 완료되어 있어야 한다.
트리거
사용자가 텍스트 질의를 입력(또는 생략)하고 현장 이미지를 첨부한 뒤 '검색' 버튼을 클릭할 때
동작
프론트엔드가 업로드된 이미지를 인코딩하여 백엔드로 전송한다.
시스템은 이미지를 VLM API로 전송하여 시각적 상황(예: 포트홀의 깊이, 안전모 미착용 등)을 분석하고 텍스트 맥락으로 추출한다.
VLM에서 추출된 '이미지 묘사 텍스트'와 사용자가 입력한 '자연어 텍스트'를 논리적으로 병합한다.
결합된 내용을 바탕으로 하이브리드 탐색 파이프라인에 태울 '최종 통합 검색 쿼리 구조체'를 생성한다.
예외 / 제한사항
오렌지파이의 대역폭 한계와 외부 API 비용을 고려하여 첨부 이미지의 용량은 최대 5MB, 확장자는 JPG/PNG로 엄격히 제한한다.
VLM API 응답 지연(Timeout) 시, 시스템은 즉각 이미지 분석을 제외하고 입력된 '텍스트' 단독 검색으로 전환(Fallback) 진행 여부를 묻는다.
비고
도로 파손, 산업현장 안전관리 등 시각적 객체 식별이 필수적인 솔루션 도메인의 특성을 반영한 핵심 데이터 전처리 단계이다.
FNC-SRC-02 · 에이전트 자율 라우팅 및 템플릿 제안
도메인: 공통 | 우선순위: 상 | 연관화면: 통합 검색 화면 | 시작 상태: 융합 쿼리 생성 완료 | 종료 상태: 라우팅 확정 또는 템플릿 제안
기능 설명: LLM 에이전트가 쿼리를 분석하여 최적의 검색 방식을 결정하고 모호한 질의를 보완할 맞춤형 템플릿을 역제안한다.
선행조건
FNC-SRC-01을 통해 텍스트로 정제된 '통합 검색 쿼리'가 생성되어 있어야 한다.
도메인별(도로, 안전, 교통) 맞춤형 질의 템플릿이 시스템에 사전 등록되어 있어야 한다.
트리거
통합 쿼리 생성이 완료되어 백엔드의 LLM 에이전트 라우팅 모듈(LangChain 등)로 데이터가 인입될 때
동작
LLM 에이전트가 텍스트 질의의 궁극적인 의도(Intent)와 도메인을 분석한다.
질의에 명확한 사실(법령 번호, 부서명 등)이나 키워드 매칭이 필요한 경우 RDBMS 중심의 검색을 지시한다.
상황 대처, 문맥 추론이 중요한 서술형 질의일 경우 Vector DB 기반 임베딩 검색을 지시한다.
질의가 너무 모호하여(예: "도로", "찾아줘") 검색이 불가능하다고 판단될 경우, 검색 실행을 즉각 보류하고 도메인에 맞는 '맞춤형 질의 템플릿(예: 유성구 포트홀 공공 민원 통계 검색)'을 화면에 역제안한다.
예외 / 제한사항
에이전트 모델의 판단 에러나 응답 지연 발생 시 무한 로딩을 방지하기 위해 기본값인 하이브리드(키워드+벡터 동시 탐색) 모드로 강제 라우팅한다.
템플릿 제안을 무시하고 사용자가 모호한 단어 검색을 강행할 경우 템플릿 역제안 로직을 1회 스킵한다.
비고
불필요한 전체 DB 스캔을 방지해 서버 자원(오렌지파이)을 보호하고, 사용자의 검색 실패(Zero-hit) 경험을 방어하는 지능형 모듈이다.
FNC-SRC-03 · 하이브리드 RAG 탐색 (캐시 무효화 적용)
도메인: 도로/안전/교통 | 우선순위: 상 | 연관화면: 검색 결과 화면 | 시작 상태: 라우팅 확정 | 종료 상태: 검색 결과 반환 완료
기능 설명: 결정된 방식에 따라 데이터를 융합 탐색하며, 매 호출마다 고유 세션을 부여해 동일 결과 반복 오류를 방지한다.
선행조건
FNC-SRC-02의 에이전트 분석을 통해 탐색 방식(라우팅 경로)과 대상 DB가 확정된 상태여야 한다.
트리거
라우터 모듈의 지시에 따라 실제 데이터베이스 및 외부 공공데이터 API 조회를 실행할 때
동작
매 검색 API 호출 시 프론트엔드와 백엔드가 고유한 세션 식별자(UUID)를 생성하여 쿼리 헤더에 부착한다.
Cache-Control: no-cache 설정을 강제 주입하여, 이전 검색 결과가 브라우저나 서버에 남아 반복 출력되는 현상을 시스템적으로 Bypass(우회)한다.
확정된 라우팅 방식에 따라 사내 문서 원본/벡터를 조회하고, 조건 충족 시 공공데이터 REST API를 호출하여 최신 수치를 가져온다.
수집된 이기종 데이터(문서 조각, 통계 JSON 등)들의 유사도 스코어와 최신성을 기준으로 재정렬(Re-ranking)을 수행하여 상위 결과를 반환한다.
예외 / 제한사항
공공데이터포털 API의 일일 트래픽 초과 혹은 서버 장애로 응답이 불가할 경우, 전체 시스템 장애로 번지지 않도록 해당 외부 데이터만 제외하고 사내 DB 검색 결과만 우선 반환한다.
오렌지파이 내부 인메모리 과부하를 방지하기 위해 한 번에 반환하고 병합하는 검색 결과 데이터 청크(Chunk)의 수는 최대 10개로 제한한다.
비고
기존 데모 버전의 가장 치명적 하자인 '캐시 동일 결과 반복 오류'를 아키텍처 레벨에서 완벽하게 차단하기 위한 필수 데이터 파이프라인이다.
FNC-DAT-01 · 사내 비정형 문서 업로드 및 파싱
도메인: 공통 | 우선순위: 상 | 연관화면: 데이터 관리 화면 | 시작 상태: 문서 업로드 대기 | 종료 상태: 텍스트 추출 완료
기능 설명: 도로 시설물 및 안전 관리 지침서(PDF/DOCX)를 업로드받아 순수 텍스트를 파싱하고 노이즈를 정제한다.
선행조건
사용자가 관리자 또는 실무자 권한을 가진 상태로 로그인되어 있어야 합니다.
오렌지파이 자체 서버의 로컬 스토리지 볼륨이 정상적으로 마운트되어 있어야 합니다.
트리거
데이터 관리 화면의 업로드 영역에 PDF 또는 DOCX 파일을 드래그 앤 드롭하거나 '파일 선택' 버튼을 통해 업로드를 확정할 때
동작
백엔드 가드 서버가 업로드된 파일의 확장자 변조 여부를 1차 검증하고 로컬 지정 경로에 파일을 저장합니다.
파이썬 파싱 라이브러리(PyPDF, python-docx 등)를 백그라운드 프로세스로 호출하여 문서 내의 날것의 텍스트를 추출합니다.
검색 및 임베딩 품질 저하를 막기 위해 공백 정제, 깨진 인코딩 문자 복원, 특수문자 필터링 등 텍스트 전처리 알고리즘을 수행합니다.
예외 / 제한사항
오렌지파이 5 Plus의 저장 공간 및 컴퓨팅 제약을 고려하여 단일 파일 업로드 용량은 최대 20MB 이하로 제한하며, 초과 시 업로드를 차단하고 안내 팝업을 노출합니다.
매크로나 악성 스크립트가 주입된 위험 파일 감지 시 즉각 파싱을 전면 중단하고 안전을 위해 파일을 영구 삭제합니다.
비고
도로 파손 매뉴얼, 스마트 팩토리 안전 지침서 등 기업 내부의 핵심 운영 데이터를 지식 베이스화하기 위한 최초의 비정형 데이터 정제 관문입니다.
FNC-DAT-02 · 시맨틱 청킹 및 로컬 Vector DB 적재
도메인: 공통 | 우선순위: 상 | 연관화면: 백엔드 처리 | 시작 상태: 텍스트 추출 완료 | 종료 상태: 임베딩 및 DB 적재 완료
기능 설명: 추출된 텍스트를 문맥 단위로 분할(Semantic Chunking)하고, 로컬 BGE-m3 모델로 벡터 변환 후 ChromaDB에 인덱싱한다.
선행조건
FNC-DAT-01 프로세스를 통해 노이즈가 제거된 순수 텍스트 데이터가 메모리 버퍼 상에 확보되어야 합니다.
엣지 서버 내에 로컬 ChromaDB(또는 FAISS) 인스턴스가 활성화되어 있어야 합니다.
트리거
문서 파싱 및 전처리 정제 공정이 성공적으로 종료되는 즉시 백그라운드 이벤트 리스너에 의해 자동 트리거됩니다.
동작
추출된 전체 텍스트의 문맥적 의미가 단절되는 것을 방지하기 위해 일반 글자 수 분할이 아닌 의미 단락 기준 분할(Semantic Chunking)을 실행하며, 청크 간 일정 비율의 오버랩(Overlap)을 부여합니다.
오렌지파이 환경에 맞춰 최적화된 로컬 경량 임베딩 모델(BGE-m3)을 호출하여 각 텍스트 청크를 다차원 벡터 배열로 변환합니다.
변환된 벡터 값을 고유 문서 ID 및 파일 메타데이터 정보와 바인딩하여 ChromaDB에 인덱싱 적재하고, 마스터 메타 정보는 PostgreSQL에 동시 동기화합니다.
예외 / 제한사항
대용량 문서 임베딩 연산 중 오렌지파이의 RAM 점유율이 임계치(90%)를 초과할 경우, OOM(메모리 초과 뻗음)을 방지하기 위해 연산을 일시 중단(Throttling)하고 가비지 컬렉터(GC)를 강제 호출한 뒤 안전 자원을 확보하고 재개합니다.
비고
단일 LLM 지식의 한계 및 단순 키워드 RAG 구조가 가진 정확도의 한계를 극복하기 위해 문맥 중심 고밀도 벡터 세팅을 수행하는 백엔드 핵심 파이프라인입니다.
FNC-PUB-01 · 공공데이터 API 카탈로그 등록 및 연동
도메인: 공통 | 우선순위: 상 | 연관화면: API 관리 화면 | 시작 상태: API 카탈로그 미등록 | 종료 상태: API 메타데이터 연동 완료
기능 설명: 공공데이터포털의 핵심 API 엔드포인트와 동적 파라미터 구조를 시스템에 등록하고 안전하게 관리한다.
선행조건
공공데이터포털(data.go.kr)에서 정상적으로 승인 및 발급받은 오픈 API 일반 인증키(API Key) 인프라가 준비되어 있어야 합니다.
트리거
