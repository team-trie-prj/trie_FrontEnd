# 백엔드 연동 질의 — API 명세(APISpec) 반영 후 확인 필요 항목

FE는 명세 기준으로 실호출 코드를 반영 완료(서버 미배포 상태라 mock 기본 유지, `VITE_USE_MOCK=false` 전환만 하면 연동됨).
아래 번호는 코드 내 `TODO(BE 확인 #N)` 주석과 일치.

| # | 항목 | 내용 | FE 임시 처리 |
|---|---|---|---|
| 1 | `/auth/kakao` 메서드 | 명세 v1=POST, v2=GET(Body 포함) 상충 | POST + `{ code, redirect_uri }` |
| 2 | `/security/prompt-check` | 요청/응답 Body 명세 미기재 | 미호출 (클라 XSS 검증만 동작) — 스키마 확정 시 검색 前 호출 추가 |
| 3 | 카탈로그 목록 | `GET /public-data` 응답 스키마 미기재 | 카탈로그 배열로 가정하고 매핑 |
| 4 | **히스토리 API 부재** | 목록/스냅샷 조회 엔드포인트가 명세에 없음 (FNC-HIS-01 복원용) | 세션 로컬 보관(새로고침 시 소실)으로 임시 구현 — `GET /history`, `GET /history/{session_uuid}` 신설 요청 |
| 5 | 공공데이터 수치 hit | `/search` hits에 공공 통계(stats) 포맷 미기재 — 예시는 vector 청크뿐 | source가 vector/bm25/rdb 외면 public_api로 간주, stats 미표시 |
| 6 | 이미지 검색 흐름 | `/search/analyze`(VLM) 후 `/search`에 unified_query 텍스트만 전달하는 2단계로 구현 — search에 image 직접 전달과 중복 여부 확인 | analyze → search(text) |
| 7 | 세션·스킵 전달 | `POST /sessions` 발급 UUID와 역제안 1회 스킵(skip) 값을 `/search`에 전달할 필드 없음 | UUID는 FE 보관·헤더(X-Session-Id)만, 스킵은 미전달 |
| 8 | 사용자 role | `/auth/kakao`·`/auth/me` user에 RBAC 권한 필드 없음 (관리자/실무자 구분 불가) | 전원 worker 처리 — role 필드 추가 요청 |
| 9 | report_type 값 | FE 양식 3종 매핑: safety-check→`inspection_log`, civil-brief→`civil_brief`, analysis→`analysis` — 서버 허용값 목록 확인 | 위 매핑 사용 |
| 10 | API 키 파라미터명 | 카탈로그 등록 시 `api_key_param` 고정값 `serviceKey` 사용 — 포털별 상이하면 입력 UI 추가 필요 | `serviceKey` 고정 |

## 연동 방식 참고 (FE 구현 기준)

- 공통 래퍼 `{ success, code, message, data }` 언랩 + `success:false` 시 `message`를 오류로 노출
- 401 → `/auth/refresh`(refresh_token 비회전) 후 1회 재시도, 실패 시 세션 종료
- 검색: `POST /sessions` → (이미지 시 `/search/analyze`, 15s 타임아웃=텍스트 단독 전환 확인) → `POST /search`(multipart) → `code=CLARIFY` 시 역제안 UI
- 보고서: `POST /reports` 단건 응답(content 마크다운) — SSE 아님 확인됨
- 업로드: `POST /documents` multipart `files`+`domain`, 삭제 `DELETE /documents/{id}`(정식 명세 확인 — 기존 협의 요청 건 해소)
- 카탈로그 등록: `POST /api-keys` → `POST /public-data/catalog` → `PATCH /public-data/{id}/fetch` 연동 테스트 → 실패 시 삭제 롤백(FNC-PUB-01 "테스트 통과 시에만 저장" 충족)
